title: ursulactl

`ursulactl` is the first interface to reach for once a cluster is running. Its verbs encode the safety properties an operator otherwise has to remember manually:

- before restarting a node, transfer every Raft group it leads to a healthy successor;
- after a restart, refuse to move on until `last_applied_index` has caught up to peers' `committed_index`;
- abort the rollout rather than corner a group with no leader.

The same orchestration is exercised under [deterministic simulation](/docs/architecture/overview), so failure modes like a leader transfer timing out, the target node entering a crash loop, or a partition during drain produce a clean abort instead of a half-rolled cluster.

## The admin plane

Nodes carry **no cluster-mutation surface on the network**. The mutating operator endpoints (raft snapshot/purge/membership/learners/leader-transfer/allow-next-revert, maintenance drain, cold-flush trigger) plus metrics are served on a separate **admin plane** bound to `server.admin_listen` — loopback (`127.0.0.1:4438`) by default. The public client plane (`:4437`) serves only stream traffic and read-only metrics.

ursulactl sends every request to the admin plane. Because that plane is loopback-bound, it reaches it through a **provider** — a named transport that knows how to both tunnel to a node's admin plane and restart it. You select one and give it a few structured values; the provider builds its own commands, so you never hand-write ssh/aws/kubectl boilerplate.

| Provider | Reaches admin via | Restarts via | Needs |
|----------|-------------------|--------------|-------|
| `direct` (default) | `admin_url` as-is (must be reachable) | — (observe only) | — |
| `ssh` | `ssh -N -L` | `ssh … systemctl restart <unit>` | `--ssh-user`, `--restart-unit` |
| `eice` | ssh over AWS EC2 Instance Connect | same as ssh | `--region`, `--ssh-user`, `--ssh-key`, `--restart-unit` |
| `command` | your `--forward-cmd` | your `--restart-cmd` | the templates below |

```bash
# SSH: nodes reachable by hostname/IP with a key
ursulactl status --config cluster.toml --provider ssh --ssh-user ec2-user

# EICE: private nodes with no inbound SSH (send-key + open-tunnel, addressed by instance id)
ursulactl status --config cluster.toml --provider eice --region us-east-1 \
  --ssh-user ec2-user --ssh-key ~/.ssh/id_ed25519

# Kubernetes / anything else: the command escape hatch
ursulactl status --config cluster.toml --provider command \
  --forward-cmd 'kubectl port-forward pod/{name} {local_port}:{admin_port}'
```

The `command` provider renders `{local_port}` `{admin_port}` `{admin_host}` `{host}` `{instance_id}` `{node_id}` `{name}` and runs under `sh -c`, the forward staying foreground for the tunnel's life (ursulactl tears it down when done). Prefer a named provider — reach for `command` only when the named set does not cover your transport.

`direct` has no way to execute a restart on a node, so it supports only the observe verbs (`status`, `wait-ready`); `restart` requires `ssh`, `eice`, or `command`.

## When to use ursulactl vs. the other surfaces

| Task | Tool |
|------|------|
| Day-2 cluster operations: restart, observe, gate on readiness | **`ursulactl`** |
| Push binaries, write systemd units, EC2 Instance Connect, S3 cleanup | `scripts/ursula_ec2.py` |
| Custom operator tooling | The admin-plane HTTP endpoints, reached over your own tunnel |

## Install

Build from the workspace alongside the server:

```bash
cargo build --release -p ursula-ctl --bin ursulactl
```

The binary lands at `target/release/ursulactl`. Drop it on your control machine; it does not need to run on the Ursula hosts themselves.

## Manifest format

Every verb accepts `--config <path>`. The manifest is **TOML, JSON, or YAML** (chosen by file extension) and carries two things: an optional `[provider]` block (how to reach the cluster) and the node list (what the cluster is). Putting the provider in the manifest keeps the command line to just the operational knobs:

```toml
[provider]
kind = "eice"
region = "us-east-1"
ssh_user = "ec2-user"
ssh_key = "~/.ssh/id_ed25519"
restart_unit = "ursula.service"

[[nodes]]
id = 1
instance_id = "i-0a1b2c3d"

[[nodes]]
id = 2
instance_id = "i-0e4f5a6b"
```

```bash
# provider lives in the file, so the command is just intent
ursulactl restart --config cluster.toml --allow-empty-raft-rejoin --ready-timeout-secs 900
```

Any `[provider]` field can be overridden on the command line (`--provider`, `--region`, `--ssh-user`, `--restart-unit`, …); flags win over the file.

Per-node fields, all optional except `id`:

- `admin_port` (default `4438`) or an explicit `admin_url` — the admin plane to reach. Tunnelling providers only need the port; `direct` needs a reachable `admin_url`.
- `instance_id` — required by `eice`; how it addresses the node.
- `host` / `ssh_host` — address for `ssh` and `{host}` interpolation. Falls back to the admin URL host, then the instance id.
- `http_url` — optional; only used for `{http_url}` in the `command` provider.

The JSON form stays compatible with `scripts/ursula_ec2.py`'s `cluster.json` (bare array, or `public_ip`/`private_ip` + `http_port`), so you can keep pointing both tools at the same file.

## `restart`

Safe rolling restart. Per target node, in manifest order (or `--only` order):

1. Snapshot every node's `/__ursula/metrics`.
2. Mark the target's [maintenance drain](/docs/operations) so it stops campaigning and re-acquiring leaderships mid-drain.
3. For every Raft group the target leads, pick the most-caught-up voter as successor and call `POST /__ursula/raft/{group}/leader/transfer/{successor}`.
4. Poll until the target leads zero groups, or abort on drain timeout.
5. Run the provider's restart action for the target (e.g. `ssh … systemctl restart <unit>`).
6. Poll until the target is back as a voter in every group and its `last_applied_index` is within `--lag-tolerance` of peers' `committed_index`, or abort on readiness timeout.
7. Clear the maintenance drain and move to the next node only on success. Any abort clears the drain mark on the current target and stops the rollout.

```bash
ursulactl restart \
  --config cluster.toml \
  --provider ssh --ssh-user ec2-user --restart-unit ursula.service \
  --drain-timeout-secs 60 \
  --ready-timeout-secs 180 \
  --lag-tolerance 16
```

The restart action comes from the [provider](#the-admin-plane); with `--provider ssh`/`eice` it is `systemctl restart <--restart-unit>`, and `--provider command --restart-cmd '…'` is the escape hatch for anything else. Add `--dry-run` to print the drain plan and skip transfers and restarts (no restart channel required, so `direct` is fine for a dry run).

Restrict the rollout with `--only`:

```bash
ursulactl restart --config cluster.toml --provider ssh --ssh-user ec2-user \
  --restart-unit ursula.service --only 2,3
```

<Warning>
The naive manual procedure of "restart followers, then leader" does **not** wait for `applied_index` to catch up between steps. Under `raft.wal.backend = "memory"` mode a target can come back as a voter while still missing committed entries, and a second restart pointed at a different node can corner the group with no live leader. `ursulactl restart` is the path that closes this gap.
</Warning>

### Restarting raft-memory nodes

On clusters running the volatile `raft.wal.backend = "memory"`, a restarted node rejoins with an **empty** log, and group leaders refuse that log reversion unless it was explicitly permitted. Two flags matter:

- `--allow-empty-raft-rejoin`: before running the restart command, ask the current leader of every group to accept one empty-log rejoin from the target. Without it the target never catches up and the readiness step times out. Do not use it on persistent-backend clusters, where an unexpected empty rejoin means data loss that deserves to be refused.
- `--ready-timeout-secs`: an amnesiac node rebuilds every group from snapshots, which for large groups takes 10+ minutes, not the default two. Size the timeout for a full rebuild.

The permission must land on the node that is leader at that moment; the endpoint answers `409 Conflict` from any other node (ursulactl resolves leadership per group and handles this). A node that is the membership initializer for some groups additionally refuses to start at all after losing its volatile log (the bootstrap-marker guard) — that recovery is an explicit operator reset on the host, outside ursulactl's reach.

## `status`

Per-node summary of Raft group count and leadership distribution, sourced from every node's `/__ursula/metrics`. Nodes whose metrics fail are reported with `metrics unavailable — …` rather than aborting the report, because `status` is meant to surface partial cluster health.

```bash
ursulactl status --config cluster.json
```

Sample output:

```
node 1 (10.0.0.1): groups=4 leaders={1: 2, 2: 2}
node 2 (10.0.0.2): groups=4 leaders={1: 2, 2: 2}
node 3 (10.0.0.3): groups=4 leaders={1: 2, 2: 2}
```

`leaders={…}` is the count of groups each node is leading from this reporter's perspective. Healthy clusters report the same distribution from every node.

## `wait-ready`

Block until every node reports `--expected-groups` Raft groups, each with a leader. Useful in CI / scripts after `start` or a config change.

```bash
ursulactl wait-ready --config cluster.json --expected-groups 4
```

Exits non-zero with a one-line reason if the timeout passes (`cluster not ready after 120s: node 3 has 1 group(s) without a leader`).

## Exit codes

| Code | Meaning |
|------|---------|
| `0` | All target nodes finished successfully (`restart`), readiness reached (`wait-ready`), or status report rendered (`status`). |
| `2` | `restart`: at least one node aborted (drain timeout, restart command non-zero, or readiness timeout). Subsequent nodes were not touched. |
| Non-zero (other) | Configuration or transport error. The error message is single-line and machine-greppable. |

## Roadmap: verbs not yet migrated

These still live in `scripts/ursula_ec2.py` because they require SSH or AWS APIs and benefit from the same EC2 Instance Connect plumbing that the script already does:

- `upload-binary`, `install-binary`: push and stage release artefacts.
- `install-service`, `install-chaos-agent`, `install-faultd`, `deploy-chaos`: write/refresh systemd units.
- `cleanup-s3`, `perf` / `perf-many`: AWS S3 and benchmark drivers.

They will move to `ursulactl` once a `NodeProvider`-backed SSH transport lands; the AWS deployment scaffolding (IAM / EC2 lifecycle / security groups) stays in Python permanently.

## Underlying HTTP surface

For custom tooling, every verb maps onto a small set of admin-plane HTTP endpoints on each node (reached over your own tunnel to `admin_listen`):

| Verb | Endpoint |
|------|----------|
| `status`, `wait-ready` | `GET /__ursula/metrics` |
| `restart` (maintenance guard) | `POST /__ursula/leadership-shed/maintenance` before the drain, `DELETE` once the node is back or the rollout aborts |
| `restart` (drain step) | `POST /__ursula/raft/{raft_group_id}/leader/transfer/{node_id}` |
| `restart` (readiness step) | `GET /__ursula/metrics` polled across all peers |

The transfer endpoint refuses with `409 Conflict` if the receiving node isn't the current leader of the group, and `400` if the target node isn't a voter. ursulactl uses this to refuse to attempt a transfer it cannot reason about.
