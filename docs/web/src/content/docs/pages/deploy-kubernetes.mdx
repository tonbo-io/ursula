---
title: Deploy on Kubernetes
description: Run Ursula on Kubernetes with the Helm chart.
---

The Ursula Helm chart starts a static-membership Raft cluster with stable
StatefulSet pod identities, durable per-pod Raft logs, a headless peer Service,
a client/admin Service, a quorum-protecting PodDisruptionBudget, and a Helm test
powered by `ursulactl`, together with soft multi-pod spread hints.

The chart is for fresh static-membership clusters. It does not perform online
Raft voter expansion, voter removal, leader handoff during Kubernetes rolling
updates, or dynamic voter reconfiguration. Those operations belong to the
future Ursula operator workflow.

## Production EKS with OpenTofu

The repository includes an OpenTofu reference stack under `deploy/eks`. It provisions a three-AZ VPC and EKS cluster, one managed node group per zone, the EBS CSI and Pod Identity add-ons, an encrypted `gp3` StorageClass, a versioned S3 bucket, and separate least-privilege identities for Ursula and the event-time indexer. It writes the complete Helm input to `generated-values.yaml` and a dedicated kubeconfig without modifying `~/.kube/config`.

Create a versioned, encrypted S3 bucket for OpenTofu state, then copy `backend.tf.example` to the ignored `backend.tf` and assign a unique state key. Copy `terraform.tfvars.example` to the ignored `terraform.tfvars`, select an explicit image tag, and restrict the EKS public API to operator or CI CIDRs; the stack rejects `0.0.0.0/0` and `::/0`. After that one-time configuration, the complete deployment path is:

```bash
cd deploy/eks
tofu init
tofu apply
KUBECONFIG=./kubeconfig helm install ursula ../../charts/ursula --namespace ursula --create-namespace -f generated-values.yaml
KUBECONFIG=./kubeconfig helm test ursula --namespace ursula
```

See [`deploy/eks/README.md`](https://github.com/tonbo-io/ursula/tree/main/deploy/eks) for input, cost, state, API CIDR, and teardown guidance. OpenTofu owns AWS prerequisites; Helm owns namespace-scoped Ursula workloads.

## Install

The chart and server images are published to GHCR on every release. One command
starts a three-voter cluster from the published artifacts:

```bash
helm install ursula oci://ghcr.io/tonbo-io/charts/ursula --version 0.2.0
```

The chart defaults to the `ghcr.io/tonbo-io/ursula` image pinned to the chart's
`appVersion`, so no image overrides are needed. The default install starts
three voters and 64 Raft groups. To use your own registry mirror, set
`global.image.repository`, `global.image.tag`, and optionally
`global.imagePullSecrets`.

## Install from a local build

To run unreleased code, build the image from the repository and install the
chart from the working tree instead:

```bash
docker build -t ursula:dev .
```

For kind:

```bash
kind load docker-image ursula:dev
```

Then point the chart at the local image:

```bash
helm install ursula charts/ursula \
  --set global.image.repository=ursula \
  --set global.image.tag=dev \
  --set global.image.pullPolicy=Never
```

## Verify readiness

```bash
helm test ursula
```

The test mounts the chart-generated `cluster-manifest.json` and runs
`ursulactl wait-ready`. It succeeds only when every configured node reports the
expected Raft group count and every group has a leader.

## Bootstrap behavior

Fresh clusters start with `raft.initMembershipPerGroup=true` so Ursula can initialize per-group Raft membership automatically. Membership initialization first checks the durable Raft log and is idempotent after a group has initialized, so persistent `logDir` deployments can keep this value enabled across restarts and upgrades. The static-cluster integration suite exercises a durable restart with per-group bootstrap still enabled. Memory-backed clusters retain their separate restart guard because they cannot recover a lost Raft log.

## Static membership

`server.replicaCount` controls the initial voter set for a fresh cluster. The
chart supports `1`, `3`, and `5`; production clusters should use `3` or `5`.

Changing `server.replicaCount` on an initialized cluster is not safe Raft voter
reconfiguration. Safe scaling requires an operator workflow that adds learners,
waits for catch-up, promotes voters, and removes old voters.

## Production topology

- Run three voters across three availability zones for the normal production profile. Each voter needs its own zonal persistent volume for the Raft log; use five voters only when tolerating two simultaneous voter failures justifies the larger write quorum.
- Configure shared S3 for cold chunks and externalized snapshots. Do not use memory Raft storage or ephemeral voter data in production.
- Run at least two stateless gateway replicas behind authenticated TLS ingress, while keeping voter and peer Services private.
- Run a fixed event-time indexer worker pool. Streams are registered dynamically, S3 is authoritative, and local caches are disposable.
- Use separate workload identities and non-overlapping S3 prefixes for the Ursula storage plane and each indexer security boundary.
- Spread replicas across zones and nodes, and alert on Raft readiness, S3 failures, indexer blocked state and lag, object growth, PVC pressure, and restarts.

The chart supplies conservative defaults and rejects several unsafe configurations, but it is not yet an operator: drain-aware voter upgrades, backups, restore drills, capacity planning, ingress authentication, and alert routing remain deployment responsibilities. The EKS OpenTofu stack supplies IAM and storage prerequisites for its reference topology.

## Cold storage

Cold storage is disabled by default. Production multi-node clusters should use
S3 or an S3-compatible object store:

```yaml
s3:
  bucket: my-ursula-bucket
  region: us-east-1
  prefix: ursula-prod

coldStorage:
  enabled: true
```

Prefer workload identity through ServiceAccount annotations instead of static
S3 credentials:

```yaml
serviceAccount:
  annotations:
    eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/ursula-s3
```

For MinIO or another S3-compatible backend, set `s3.endpoint`.

## Event-time indexer

`ursula-indexer` is an optional cluster-level worker pool rather than a Raft background worker. Pod count is independent of stream count. Keeping it outside voter processes prevents query, Parquet, and compaction work from affecting stream commit latency and lets every projection be rebuilt independently.

```yaml
s3:
  bucket: my-index-bucket
  region: us-east-1

indexer:
  enabled: true
  replicaCount: 2
  s3:
    prefix: indexes
  serviceAccount:
    annotations:
      eks.amazonaws.com/role-arn: arn:aws:iam::123456789012:role/ursula-index
```

After creating an `application/json` source, dynamically register it with `PUT /v1/indexes/{id}` on the internal indexer Service. Registration verifies `json-record-coordinates-v1` and stores the current retained boundary as `indexed_from_record`; existing streams do not need to retain record 0. No Helm upgrade is needed. Workers claim `(stream, record range)` tasks across all registrations, so multiple pods can process a hot stream while one pod can share time across small streams. Range completion may be out of order, but the durable watermark advances only across a gap-free prefix starting at `indexed_from_record`. Expiring S3 claims reduce duplicate work while immutable parts and manifest CAS provide correctness.

The indexer Service is an internal ClusterIP. If remote applications need `/v1/indexes/{id}/events`, put an authenticated, path-aware proxy in front of it and do not expose registration, deletion, or `POST /v1/indexes/{id}/status/resume` administration routes.

## Operational endpoints

Ursula exposes a single admin endpoint on the client plane:

- `GET /__ursula/metrics`: per-node JSON snapshot with Raft group state, hot
  bytes, cold backpressure counters, and per-core mailbox depth.

There are no dedicated `/__ursula/healthz` or `/__ursula/readyz` probes yet.
Server HTTP probes are disabled in the current chart, and cluster-level
readiness should come from `helm test` or `ursulactl wait-ready`.

The mutating operator surface (raft operations, maintenance drain, cold-flush
trigger) lives on the admin plane bound to pod loopback (`127.0.0.1:4438`);
reach it with `kubectl port-forward <pod> 4438:4438`.

On pod termination the server handles SIGTERM by draining its listeners:
it stops accepting, finishes in-flight requests, and exits 0, forcing exit
after 20 seconds if a request hangs. Keep `terminationGracePeriodSeconds`
above that (the Kubernetes default of 30s is fine). For zero-disruption
rollouts, still drain leaderships first with `ursulactl restart`.

## Access locally

```bash
kubectl port-forward svc/ursula 4437:4437
curl http://127.0.0.1:4437/__ursula/metrics
```

## Upgrade limits

Until the operator exists, Kubernetes StatefulSet rolling updates do not
transfer leaders, coordinate applied-index catch-up, or mutate Raft membership.
Use `ursulactl restart` manually for drain-aware rolling restarts when you need
operationally safe restarts on an initialized cluster.
